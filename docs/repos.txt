### OPCAO 1: MONOREPO (TUDO EM UM UNICO REPOSITORIO) ###

Estrutura:
multilingual-code/
  +-- packages/
  |   +-- core/
  |   +-- csharp-adapter/
  |   +-- python-adapter/
  |   +-- vscode-extension/
  +-- languages/
  +-- examples/
  +-- docs/

VANTAGENS:
+ Mudancas atomicas (um PR pode mudar core + adapter + tabelas)
+ Versionamento unificado e simples
+ CI/CD mais simples de configurar
+ Facil garantir compatibilidade entre componentes
+ Menos overhead de manutencao no inicio
+ Clone unico tem tudo
+ Monorepo tools (Nx, Turborepo) podem ajudar

DESVANTAGENS:
- Repositorio cresce muito com o tempo
- Clone pesado (tudo junto)
- Dificil para contribuidores que so querem adicionar traducoes
- Build pode ser lento
- Dificil para usuarios que so querem um adapter especifico
- Permissoes de acesso sao tudo-ou-nada
- Historico Git fica confuso (mudancas em tudo misturado)

QUANDO USAR:
- Projeto pequeno/medio
- Equipe pequena e coesa
- Componentes muito acoplados
- Inicio de projeto (MVP)

### OPCAO 2: MULTI-REPO (SEPARADO POR COMPONENTE) ###

Estrutura:
multilingual-code-core/           (C# engine)
multilingual-code-csharp/         (CSharp adapter)
multilingual-code-python/         (Python adapter)
multilingual-code-vscode/         (VS Code extension)
multilingual-code-visualstudio/   (Visual Studio extension)
multilingual-code-intellij/       (IntelliJ plugin)
multilingual-translations/        (tabelas de idiomas)

VANTAGENS:
+ Contribuicoes muito focadas
+ Clone leve para contribuidores especificos
+ Versoes totalmente independentes
+ Times diferentes podem manter repos diferentes
+ Traducoes evoluem independentemente do core
+ Facil dar permissoes granulares
+ Cada repo tem seu proprio CI/CD otimizado
+ Historico Git limpo por componente

DESVANTAGENS:
- Sincronizacao complexa entre repositorios
- Versionamento entre repos e muito complicado
- Breaking changes sao um pesadelo
- CI/CD precisa orquestrar multiplos repos
- Duplicacao de configuracao (.github, etc)
- Overhead de manutencao alto
- Dificil para novos contribuidores entenderem a estrutura
- Testes de integracao entre repos sao complexos

QUANDO USAR:
- Projeto muito grande e maduro
- Equipes distribuidas e independentes
- Componentes totalmente desacoplados
- Necessidade de releases independentes frequentes

### OPCAO 3: HIBRIDO (CORE + PLUGINS SEPARADOS) ###

Estrutura:
multilingual-code/                 (PRINCIPAL)
  +-- packages/
  |   +-- core/
  |   +-- csharp-adapter/
  |   +-- vscode-extension/
  +-- examples/
  +-- docs/

multilingual-translations/         (SEPARADO)
  +-- programming-languages/
  +-- natural-languages/
  +-- templates/
  +-- tools/

[FUTUROS - conforme necessario]
multilingual-code-python/
multilingual-code-visualstudio/
multilingual-code-intellij/

VANTAGENS:
+ Balanco entre simplicidade e modularidade
+ Traducoes podem ser contribuidas facilmente SEM tocar codigo
+ Core + MVP mantem sincronizado
+ Releases de traducoes independentes (sem afetar core)
+ Clone do repo principal tem tudo essencial
+ Facil expandir depois
+ Contribuidores de traducoes nao precisam entender C#
+ Menos overhead que multi-repo total

DESVANTAGENS:
- Ainda precisa sincronizar core <-> traducoes
- Decisoes sobre o que vai em qual repo
- Dois repos para gerenciar (minimo)

QUANDO USAR:
- Projeto em crescimento
- Quer facilitar contribuicoes de traducoes
- MVP precisa estar coeso
- Planejando expansao futura

ESTRATEGIA DE VERSIONAMENTO E COMPATIBILIDADE
================================================================================

### VERSIONAMENTO PRINCIPAL (multilingual-code) ###

Semantic Versioning: MAJOR.MINOR.PATCH

MAJOR (v1.0.0 -> v2.0.0):
- Breaking changes no Core API
- Breaking changes na interface ILanguageAdapter
- Mudancas que exigem re-escrever adapters

MINOR (v1.0.0 -> v1.1.0):
- Novas features
- Novos adapters incluidos
- Melhorias de performance
- Features opcionais adicionadas

PATCH (v1.0.0 -> v1.0.1):
- Bug fixes
- Correcoes de documentacao
- Pequenas otimizacoes

### VERSIONAMENTO TRADUCOES (multilingual-translations) ###

Semantic Versioning independente: MAJOR.MINOR.PATCH

MAJOR (v1.0.0 -> v2.0.0):
- Mudanca de esquema JSON
- Reestruturacao completa
- Breaking changes que exigem update do core

MINOR (v1.0.0 -> v1.1.0):
- Novo idioma natural adicionado
- Nova linguagem de programacao adicionada
- Expansao de common-identifiers

PATCH (v1.0.0 -> v1.0.1):
- Correcoes em traducoes existentes
- Pequenas melhorias
- Adicao de keywords faltantes

### ARQUIVO DE COMPATIBILIDADE ###

No multilingual-translations, criar compatibility.json:

{
  "translations": {
    "version": "1.5.0",
    "compatibleWith": {
      "multilingual-code": ">=1.0.0 <2.0.0"
    }
  },
  "languages": {
    "pt-br": {
      "version": "1.2.0",

    },
    "es-es": {
      "version": "1.1.0",
      "csharp": "complete",
      "python": "none"
    }
  }
}

PROCESSO DE RELEASE
================================================================================

### RELEASE DO CORE (multilingual-code) ###

1. Criar branch release/vX.Y.Z
2. Atualizar versoes em:
   - package.json (root)
   - MultiLingualCode.Core.csproj
   - package.json (vscode)
3. Atualizar CHANGELOG.md
4. Rodar todos testes
5. Build de release
6. Criar tag vX.Y.Z
7. Push tag -> CI automaticamente:
   - Publica NuGet package
   - Publica VS Code extension
   - Cria GitHub Release
8. Merge release -> main
9. Merge release -> develop

### RELEASE DE TRADUCOES (multilingual-translations) ###

1. Novo idioma adicionado via PR
2. PR aprovado e merged em main
3. Atualizar version em package.json
4. Criar tag vX.Y.Z
5. Push tag -> CI automaticamente:
   - Publica NPM package
   - Publica NuGet package
   - Atualiza compatibility.json
   - Cria GitHub Release
6. Opcional: Notificar repo principal (issue/PR)

IMPORTANTE: Releases de traducoes NAO exigem release do core!

### CI/CD - multilingual-code ###

.github/workflows/ci.yml:
- Trigger: Push em qualquer branch
- Jobs:
  1. Build Core (dotnet build)
  2. Test Core (dotnet test)
  3. Build VS Code Extension (npm run compile)
  4. Test Extension (npm test)
  5. Lint (eslint, dotnet format)
  6. Coverage report
- Artifacts: nenhum

.github/workflows/release.yml:
- Trigger: Tag vX.Y.Z
- Jobs:
  1. Build Release Core
  2. Package NuGet
  3. Publish NuGet
  4. Build Release Extension
  5. Package VSIX
  6. Publish VS Code Marketplace
  7. Create GitHub Release
- Artifacts: NuGet package, VSIX file

### CI/CD - multilingual-translations ###

.github/workflows/validate.yml:
- Trigger: Pull Request
- Jobs:
  1. Validate JSON schema
  2. Check completeness
  3. Check duplicates
  4. Generate coverage report
  5. Comment on PR with results
- Artifacts: Coverage report

.github/workflows/publish.yml:
- Trigger: Tag vX.Y.Z
- Jobs:
  1. Validate all tables
  2. Package NPM
  3. Publish NPM
  4. Package NuGet
  5. Publish NuGet

  7. Update compatibility matrix
- Artifacts: NPM package, NuGet package

================================================================================
PERGUNTAS FREQUENTES
================================================================================

Q: Por que nao colocar traducoes no repo principal?
A: Para facilitar contribuicoes. Tradutores nao precisam entender C#, 
   clonar um repo gigante, ou esperar release do core.

Q: Como garantir compatibilidade entre repos?
A: Arquivo compatibility.json, versionamento semantico, e CI/CD que 
   valida compatibilidade.

Q: E se uma traducao depende de uma feature nova do core?
A: Traducoes especificam versao minima do core em metadata. Core pode
   ter features opcionais que traducoes nao precisam suportar.

Q: Quando separar um adapter em repo proprio?
A: Quando: (1) ficar muito complexo, (2) equipe separada mantendo,
   (3) ciclo de release muito diferente.

Q: Como fazer release coordenado core + traducoes?
A: Normalmente nao precisa. Mas se precisar: release core primeiro,
   entao release traducoes referenciando nova versao.
